# AI 托管真实历史数据待办

状态：Go 后端的 MySQL 真实历史、强制 HOLD、决策审计和只读历史 API 已实现；前端统一数据源仍待处理。

## 当前状态

- 后端 AI 托管引擎定期主动读取链上博弈池信息、虚拟储备量、IPFS 元数据和黄金行情。
- 后端按合约 `NO, YES` 储备顺序计算真实 YES/NO 占比，合并 IPFS 有效种子，并把链上采样持久化到 MySQL。
- AI 提示词包含当前占比、真实历史、原始储备、池信息、详细说明和黄金行情。
- 有效历史少于 3 点时后端强制 HOLD，不调用模型、不发送交易。
- `GET /api/gold/market-history` 提供按时间升序排列的只读真实历史。
- 规则 HOLD、模型决策、跳过原因和交易哈希写入 `ai_decisions` 审计表。
- 新版前端可以读取 IPFS 元数据中的 `history`：`[{t|time, y|yes, n|no}]`。
- 前端创建博弈池时尚未写入 `history`；IPFS 缺少历史时，前端会随机生成模拟曲线。这部分必须移除或替换，不能作为 AI 投研依据。

## 目标行为

1. 后端按固定周期从链上读取每个被托管博弈池的储备量。
2. 按合约真实定价规则计算 YES/NO 占比，记录带时间戳的历史点。
3. AI 每轮分析静态 IPFS 元数据、当前链上占比、真实历史序列和黄金行情。
4. AI 只能输出 `buy_yes`、`buy_no` 或 `hold`。
5. 历史数据不足时后端强制覆盖为 `hold`，不允许下单。

## 演示后剩余工作

- 前端折线图改读后端 `market-history` API。
- 移除前端 `generateMockHistory`，让前端展示与 AI 研究使用同一份 MySQL 真实历史。
- 根据生产数据量设计历史归档或保留周期；当前数据库保留完整历史，AI 每轮只读取配置上限。
- 如需后端重启后自动恢复托管，另行设计钱包授权或稳定主密钥；私钥不得明文写入 MySQL。

## 演示前约束

- 不改动现有托管启停协议。
- 不改动现有买入动作、置信度、冷却和固定金额规则，只在前面增加历史门禁。
- 不把前端随机模拟曲线送入 AI。
